# Agent Team Monitor 游戏化办公场景设计文档

**日期**: 2026-03-08
**版本**: 1.0
**状态**: 已批准

## 概述

将 Agent Team Monitor 的 Web 界面改造为游戏化的办公场景视图，使用 Phaser 3 游戏引擎渲染一个动态的、类似模拟经营游戏的办公室环境。Agent 以员工形式呈现，可以看到他们在办公室内工作、移动、去洗手间、健身等活动。

## 设计目标

1. **娱乐化监控体验** - 将枯燥的数据监控转变为有趣的可视化场景
2. **真实办公场景** - 模拟真实公司的办公室布局和员工行为
3. **动态扩展** - 根据团队数量自动调整场景大小和布局
4. **易于扩展** - 支持未来添加图片素材，无需修改核心代码
5. **性能优化** - 支持大规模团队（10+ 团队，50+ agents）流畅运行

## 核心决策

### 界面集成方式
- **完全替换 Web 界面** - 将整个 Web 模式改造为游戏化视图
- 保留 TUI 模式不变
- 移除现有的列表式界面

### 场景缩放策略
- **固定元素大小，场景动态扩展** - 团队多时扩大场景范围
- 用户可通过鼠标滚轮缩放、拖拽查看
- 初始视角自动适配所有团队

### Agent 状态映射
- **位置 + 视觉效果结合**
  - 位置：工位=工作中，功能区=休息/活动
  - 视觉：颜色（绿色=busy，灰色=idle）+ 动画（呼吸、移动速度）
  - 气泡：显示实时活动（💭思考、🔧工具、📨消息）

### 技术选型
- **Phaser 3 游戏引擎** - 成熟、功能丰富、资源管理完善
- **原生 JavaScript** - 保持与现有代码风格一致
- **Canvas 2D 渲染** - 初期使用简笔画，支持图片热替换

```

  ↓ (增量更新)
```

### 核心类设计

**OfficeScene** - 主场景
- 管理所有团队、agent、功能区
- 处理数据更新和布局重排
- 协调各个系统（资源、动画、相机）

**Agent** - Agent 精灵
- 状态机：Busy / Idle / GoingOut
- 移动动画、状态切换动画
- 气泡显示（活动提示）

**Team** - 团队管理
- 管理团队内所有 agent
- 办公室布局（2列多行工位）
- 团队信息显示

**LayoutManager** - 布局管理器
- 动态生成不规则办公室布局
- 根据团队数量调整场景大小
- 公共设施放置算法

**ActivitySimulator** - 活动模拟器
- 模拟 idle agent 的随机活动
- 寻路算法（A* 或简化版）
- 碰撞检测和避让

## 办公场景布局设计

### 布局理念

- **有机不规则** - 避免死板网格，像真实租用的办公空间
- **动态成长** - 人少时紧凑，人多时逐步扩张
- **老板办公室在走廊尽头** - 象征权威位置

### 规模分级

#### 小公司（1-2 个团队，≤10 人）

```
│    │  👤  👤        │

│ ☕ │ Team B 室 │🏢老板│
│    │  👤  👤  │    │
└─────────────────────┘
```

- 场景尺寸：800x600px
- 简单 L 型或 U 型布局
- 公共设施：洗手间 + 茶水间

- 不规则排列

### 布局算法

#### 优先级排列
1. 洗手间（左上角固定）
2. 团队办公室（按创建时间顺序）
3. 老板办公室（走廊尽头）
4. 其他设施（填充空隙）

#### 放置规则
- 从左上角开始，沿墙壁顺时针放置
- 每个房间随机偏移 ±20px（打破网格感）
- 房间尺寸根据成员数：
  - 1-2 人：200x150px
  - 3-4 人：250x180px
  - 5-6 人：300x220px
  - 7+ 人：350x260px

#### 走廊生成
- 自动计算房间之间的空隙作为走廊
- 保证所有房间可达
- 老板办公室放在最长走廊的尽头

#### 场景边界
- 初始：800x600px
- 每增加 2 个团队，宽度 +300px 或高度 +250px
- 最大：2400x1600px

### 办公室内部布局（2列多行）

```
门
↓
👤  👤  ← 第1行
👤  👤  ← 第2行
👤  👤  ← 第3行
```

- 列间距：120px
- 行间距：80px
- 门朝向中央走廊
- 顶部显示团队名称标签

### 公共设施动态添加

- 1-2 团队：洗手间 + 茶水间
- 3-4 团队：+ 健身区
- 5-6 团队：+ 会议室
- 7+ 团队：+ 档案室、打印区等

## Agent 行为与动画系统

### 状态机

```
┌─────────┐  任务完成   ┌─────────┐  30%概率   ┌─────────────┐
│  Busy   │ ────────→  │  Idle   │ ────────→  │ Going Out   │
│ (工位)  │            │ (工位)  │            │ (去功能区)   │
└─────────┘            └─────────┘            └─────────────┘
     ↑                      ↑                        │
     │      新任务          │      70%概率留在办公室   │
     │                      │                        │
     └──────────────────────┴────────────────────────┘
                         返回工位
```

### 状态详细定义

#### Busy（工作中）
- **位置**: 固定在办公室内的工位上
- **视觉**: 绿色圆点，轻微呼吸动画（缩放 0.95-1.05）
- **气泡**: 随机显示 💭（思考）、🔧（工具调用）、📨（消息）
- **行为**: 不会离开工位

#### Idle（空闲）
- **位置**: 在办公室内随机走动，或坐在工位上
- **视觉**: 灰色圆点，慢速呼吸动画
- **行为决策**（每 10-30 秒触发）：
  - 70% 概率：在办公室内随机走动（速度 50px/秒）
  - 30% 概率：走出办公室去功能区

#### Going Out（外出）
- **目标选择**（随机权重）：
  - 洗手间：40%（停留 20-40 秒）
  - 茶水间：35%（停留 15-30 秒）
  - 健身区：15%（停留 30-60 秒）
  - 其他团队办公室：10%（串门，停留 10-20 秒）
- **移动**: 使用 A* 寻路，速度 100px/秒
- **到达后**: 在功能区内随机位置停留，然后原路返回

### 动画细节

#### 移动动画
- 使用 Phaser Tween 系统实现平滑移动
- 缓动函数：Cubic.InOut（自然加减速）
- 转身：agent 圆点内绘制简单的"眼睛"，朝向移动方向
- 路径：贝塞尔曲线（避免机械直线）

#### 状态切换动画
- Busy → Idle：颜色渐变（绿→灰，0.5 秒）
- Idle → Busy：颜色渐变 + 轻微跳跃（表示"开始干活"）
- 离开工位：淡出 → 移动 → 淡入

#### 气泡系统
- 根据 API 返回的 activity 数据显示
- 气泡样式：白色圆角矩形，emoji 居中
- 显示时长：2 秒后淡出
- 多个活动：队列显示，不重叠

#### 碰撞避让
- Agent 之间保持 20px 最小距离
- 移动时检测前方是否有其他 agent
- 如果碰撞：稍微偏移路径或等待 1 秒

### 简笔画绘制（临时素材）

#### Agent 绘制
```javascript
// 圆形身体 + 简单表情
circle(x, y, 20, color);           // 身体
circle(x-8, y-5, 3, '#000');       // 左眼
circle(x+8, y-5, 3, '#000');       // 右眼
arc(x, y+5, 10, 0, Math.PI);       // 微笑
text(agentName, x, y+30);          // 名字
```

#### 办公室绘制
```javascript
// 墙壁 + 门 + 窗户
rect(x, y, width, height, '#E8E8E8');      // 墙壁
rect(doorX, doorY, 40, 60, '#8B4513');     // 门
rect(windowX, windowY, 60, 40, '#87CEEB'); // 窗户
text(teamName, x+10, y+20);                // 团队名
```

#### 功能区绘制
- **洗手间**: 矩形 + 🚻 emoji + 门轮廓
- **茶水间**: 矩形 + ☕ emoji + 桌子简笔画
- **健身区**: 矩形 + 💪 emoji + 哑铃图标
- **老板办公室**: 矩形 + 🏢 emoji + 豪华门

## 资源管理与图片替换系统

### 设计目标

1. **开发期无需等待美术资源** - 使用 Canvas 绘制函数作为 fallback
2. **图片替换零代码改动** - 只需放入图片文件即可
3. **支持部分替换** - 可以只替换部分元素（如 agent），保留其他简笔画
4. **降级优雅** - 图片加载失败自动回退到绘制函数

### 资源配置文件

`web/static/js/game/config/assets.json`:

```json

    "default": { "type": "draw", "fallback": "drawAgent" },

    "idle": { "type": "image", "path": "assets/agent-idle.png", "fallback": "drawAgentIdle" }
  },
  "rooms": {
    "office": { "type": "image", "path": "assets/office-room.png", "fallback": "drawOffice" },
    "boss": { "type": "image", "path": "assets/boss-room.png", "fallback": "drawBossRoom" }
  },
  "facilities": {
    "restroom": { "type": "image", "path": "assets/restroom.png", "fallback": "drawRestroom" },
    "cafe": { "type": "image", "path": "assets/cafe.png", "fallback": "drawCafe" },
    "gym": { "type": "image", "path": "assets/gym.png", "fallback": "drawGym" }
  },
  "decorations": {
    "plant": { "type": "image", "path": "assets/plant.png", "fallback": "drawPlant" },

  }

```

### ResourceManager 类

核心功能：
- 初始化时尝试加载所有图片
- 图片加载失败时标记使用 fallback
- 提供统一的资源获取接口
- 注册和管理 Canvas 绘制函数

### 使用流程

#### 初期（无图片）
```javascript
const asset = resourceManager.getAsset('agents', 'busy');
// 返回: { type: 'draw', func: drawAgentBusy }
// 使用 Canvas 绘制函数渲染
```

#### 添加图片后
```javascript
// 1. 将图片放入 web/static/assets/agent-busy.png
// 2. 刷新页面，ResourceManager 自动检测并加载
// 3. 同样的代码现在返回:
const asset = resourceManager.getAsset('agents', 'busy');
// 返回: { type: 'image', key: 'agents_busy' }
// 使用 Phaser Sprite 渲染
```

### 绘制函数库

所有 fallback 绘制函数集中在 `DrawFunctions.js`：

```javascript
export const drawFunctions = {
  drawAgent: (graphics, x, y, color) => { /* ... */ },
  drawOffice: (graphics, x, y, width, height) => { /* ... */ },
  drawRestroom: (graphics, x, y) => { /* ... */ },
  // ... 其他绘制函数
};
```

### 图片规格建议

- **Agent**: 64x64px PNG，透明背景
- **办公室**: 300x220px PNG，包含墙壁和门
- **功能区**: 150x120px PNG，透明背景
- **装饰物**: 32x32px 或 64x64px PNG

## 数据同步与性能优化

### 数据更新策略

#### 轮询机制
- **频率**: 1 秒轮询一次 `/api/state`
- **增量更新**: 只处理变化的部分（团队增删、agent 状态变化）
- **差异检测**: 对比新旧状态，生成变更列表

#### 动画与数据分离
- **数据更新**: 1 秒一次（API 轮询）
- **动画循环**: 60 FPS（Phaser 游戏循环）
- **Agent 移动**: 独立于数据更新，平滑过渡 3-5 秒

### 性能优化

#### 对象池
```javascript
class AgentPool {
  // 预创建 50 个 Agent 对象
  // 避免频繁创建/销毁导致的 GC 压力
}
```

#### 视锥剔除
- 只渲染可见区域内的元素
- 屏幕外的 agent 设置为不可见，跳过更新

#### 批量渲染
- 使用 RenderTexture 缓存静态元素（墙壁、家具）
- 只重绘动态元素（agent、气泡）

#### LOD（细节层次）
- 距离相机 > 800px 的 agent 简化渲染（只显示圆点，不显示表情）
- 团队数 > 10 时降低非关键动画帧率到 30 FPS

#### 内存管理
- 限制活动气泡数量（每个 agent 最多 1 个）
- 定期清理已完成的 tween 动画
- 移除屏幕外的装饰元素

### 错误处理

#### API 失败降级
- 超时 5 秒后标记为断线
- 保持当前状态，继续动画
- 显示连接错误提示

#### 数据验证
- 验证 API 返回数据结构
- 无效数据时使用上一次有效状态

### 调试工具

- **F12**: 显示调试面板（FPS、agent 数量、房间数量、内存使用）
- **F11**: 切换碰撞边界显示
- **F10**: 暂停/恢复 agent 移动

## 技术风险与缓解

### 风险 1：Phaser 3 学习曲线
- **缓解**: 使用官方示例和文档，只使用基础功能
- **备选**: 如果遇到严重问题，可降级到 PixiJS

### 风险 2：大规模场景性能
- **缓解**: 提前实现性能优化（对象池、视锥剔除）
- **备选**: 限制最大团队数量（如 20 个）

### 风险 3：寻路算法复杂度
- **缓解**: 使用简化的寻路（直线 + 避障），而非完整 A*
- **备选**: 预定义走廊路径，agent 沿路径移动

### 风险 4：浏览器兼容性
- **缓解**: 使用 Phaser 的 Canvas 渲染器（兼容性好）
- **备选**: 提供降级提示，建议使用现代浏览器

## 总结

本设计方案将 Agent Team Monitor 从传统的列表式监控界面转变为一个生动有趣的游戏化办公场景。通过 Phaser 3 游戏引擎，我们可以实现流畅的动画、灵活的布局和丰富的交互。资源管理系统确保了开发期可以使用简笔画快速迭代，未来可以无缝替换为精美的图片素材。

核心设计原则：
1. **真实感** - 模拟真实办公室的布局和员工行为
2. **动态性** - 根据团队规模自动调整场景
3. **可扩展** - 易于添加新功能和素材
4. **高性能** - 支持大规模场景流畅运行

该方案已获得用户批准，可以进入实现阶段。
